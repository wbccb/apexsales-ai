# 🚀 销售 AI 闭环系统 —— v0.0.2 需求文档（补齐版）

## 一、文档目标
- 定义 v0.0.2 的增量需求，补齐 v0.0.1 后仍缺失的关键能力。
- 优先级严格执行：**RAG（P0） > 真实落库（P1） > 体验增强（P2）**。
- 目标是把系统从“可演示”推进到“可持续使用、可追溯”。

## 二、范围定义

### In Scope（v0.0.2 交付范围）
- P0：RAG 知识库能力（PDF 上传、切片向量化、检索注入总结/PRD）。
- P1：核心业务数据真实持久化（替换内存态存储）。
- P2：分享体验与合同结构化增强（直开分享页、报价字段入合同）。

### Out of Scope（v0.0.2 不包含）
- 多租户权限体系、复杂组织架构与审计策略。
- 生产级高可用架构（多活、跨地域容灾、自动扩缩容）。
- 多语种专业术语精调与模型训练平台化。

## 三、功能需求

### FR-201 RAG 文档上传与归档（P0）
- 系统支持上传产品文档与报价单 PDF。
- 系统对上传文档记录基础元数据（名称、归属、状态、时间）。
- 系统支持按业务维度查询可用知识文档列表。

### FR-202 文档切片与向量化（P0）
- 系统可对 PDF 文档进行切片并生成 embedding。
- 系统将切片内容、页码、token 数与向量索引关联存储。
- 系统支持失败重试与状态标记（待处理/处理中/完成/失败）。

### FR-203 检索增强总结（P0）
- 会话总结前，系统基于 transcript 做向量检索并返回 TopK 片段。
- 生成 PRD 时，系统把检索到的报价/产品信息注入 Prompt。
- 输出需包含可追溯来源信息，避免无依据报价。

### FR-204 核心实体真实落库（P1）
- 系统将 `users/sessions/utterances/prds/pocs/contracts/voice_profiles` 持久化。
- 系统保证现有 API 路由保持兼容，调用方无感升级。
- 系统支持服务重启后历史数据可恢复查询。

### FR-205 分享与下载持久化映射（P1）
- 系统将 `share_uuid -> poc` 映射持久化，避免重启失效。
- 系统将合同记录与文件引用持久化，支持按 contract_id 稳定下载。

### FR-206 分享体验增强（P2）
- 前端提供独立分享页路由，URL 携带 UUID 可自动加载并渲染 POC。
- 客户打开链接后无需手动输入 UUID。

### FR-207 合同结构化报价增强（P2）
- 合同模板支持结构化报价字段注入（单价、数量、税率、总价、付款节点）。
- 合同内容中的报价字段与检索/PRD 结果保持一致与可追溯。

## 四、非功能需求
- 可追溯性：RAG 结果必须可定位到文档与切片来源。
- 可靠性：文档处理链路支持失败重试与状态可观测。
- 一致性：持久化后接口响应结构与 v0.0.1 保持兼容。
- 性能：P0 阶段优先保证检索准确性与可用性，后续再优化吞吐与延迟。
- 安全性：密钥配置走环境变量；下载接口校验资源存在与归属关系。

## 五、验收标准

### P0 验收（RAG）
- 上传 1 份产品文档与 1 份报价 PDF 后，系统完成向量化且可检索。
- 生成总结时，PRD 内出现“匹配报价/来源依据”段落。
- 任意引用内容可追溯到具体文档与片段，不出现无来源报价。

### P1 验收（真实落库）
- 服务重启后，历史 sessions/utterances/prds/pocs/contracts 仍可查询。
- `/poc/{uuid}` 在重启后可持续访问原始 POC 内容。
- 合同下载可按 contract_id 稳定访问并保持归属可追踪。

### P2 验收（体验增强）
- 客户打开分享链接可直接看到 POC，无需手动输入 UUID。
- 合同 PDF 显示结构化报价字段，字段值与 PRD/知识来源一致。

## 六、优先级与实施顺序
- Step 1（P0）：先交付 RAG 最小闭环（上传 -> 切片向量化 -> 检索 -> 总结注入）。
- Step 2（P1）：完成核心实体持久化与现有接口无感迁移。
- Step 3（P2）：补齐分享直开体验与合同结构化报价渲染。
