# 销售 AI 闭环系统 v0.0.2 —— 详细设计方案

## 1. 设计目标
- 对齐 FR-201~FR-207，给出可直接开发的模块、数据模型、接口、流程与迁移方案。
- 保证 v0.0.1 既有接口兼容，不影响前端现有链路。
- 以“先 P0、后 P1、再 P2”执行，避免范围扩散。

## 2. 系统组件与目录建议

### 2.1 后端组件
- `app/api`：路由定义与请求响应模型。
- `app/services`：业务服务（knowledge/summary/prd/poc/contract/session）。
- `app/repositories`：数据库访问层。
- `app/tasks`：异步任务执行（切片、embedding、重建索引）。
- `app/integrations`：LLM、向量检索、对象存储客户端封装。

### 2.2 前端组件
- `app/(dashboard)/knowledge`：文档上传与状态页。
- `app/share/[uuid]`：分享直开页。
- `components/summary`：总结与引用来源展示。
- `components/contract`：结构化报价预览与导出入口。

## 4. 状态机设计

### 4.1 knowledge_documents.status
- pending：文档已上传，待处理。
- processing：切片与向量化中。
- ready：可用于检索。
- failed：处理失败，可重试。

### 4.2 状态流转规则
- 上传成功：`pending`
- 任务启动：`pending -> processing`
- 处理成功：`processing -> ready`
- 处理失败：`processing -> failed`
- 手动重建：`failed|ready -> processing`

## 5. 接口详细设计

### 5.1 P0 新增接口

#### POST /knowledge/upload
- 请求：`multipart/form-data`
  - file: PDF
  - owner_user_id: string（可选）
  - business_tag: string（可选）
- 响应：
  - document_id
  - status
  - filename

#### GET /knowledge/documents
- 请求参数：
  - status（可选）
  - owner_user_id（可选）
  - page / page_size
- 响应：
  - list[document]
  - pagination

#### POST /knowledge/reindex/{document_id}
- 请求：空体
- 响应：
  - document_id
  - status=processing

#### POST /knowledge/retrieve
- 请求：
  - query: string
  - top_k: int
  - business_tag: string（可选）
- 响应：
  - matches: [{document_id, chunk_id, score, page_no, content}]

### 5.2 P0 改造接口

#### POST /session/{session_id}/summary
- 新增请求参数：
  - rag_enabled: boolean（默认 true）
  - top_k: int（默认 5）
- 新增响应字段：
  - citations: [{document_id, chunk_id, score, page_no, snippet}]
  - rag_used: boolean
- 兼容要求：
  - 保留 `prd_id` 与 `markdown` 字段，前端旧逻辑可直接工作。

### 5.3 P1 兼容接口
- v0.0.1 全量接口路径与主字段不变，仅内部从内存态改为数据库访问。
- `/poc/{share_uuid}` 与 `/contract/{contract_id}/download` 读取持久化映射。

### 5.4 P2 新增前端路由
- `GET /share/{uuid}`（前端页面路由）
  - 页面加载即调用 `/poc/{uuid}`。
  - 成功渲染 Sandpack，失败显示错误与返回入口。

## 6. 核心流程详设

### 6.1 上传与向量化流程（异步）
1. `POST /knowledge/upload` 保存文件到对象存储，写 `knowledge_documents(pending)`。
2. 任务队列触发 `index_document(document_id)`。
3. 读取 PDF，按段落+长度切片，生成 chunk。
4. 调 embedding 接口，写入 `knowledge_chunks`。
5. 更新文档状态为 `ready`，失败则 `failed + error_message`。

### 6.2 总结增强流程
1. 拉取 session utterances，拼 transcript。
2. transcript 检索 top_k chunks。
3. 构造增强 Prompt：`transcript + retrieved_context + citation_rules`。
4. 生成 markdown，保存 prd。
5. 保存 citations 到 `prd_citations`。

### 6.3 合同结构化报价流程
1. 读取 PRD 与引用来源中的报价条目。
2. 映射为标准 `quote_payload`。
3. 模板渲染：基础条款 + 报价表格字段。
4. 输出 PDF，写 `contracts` 并返回下载地址。

## 7. 算法与参数建议
- 切片策略：
  - chunk_size: 500~800 tokens
  - chunk_overlap: 80~120 tokens
- 检索策略：
  - first_stage: 向量召回 top_k=12
  - rerank_stage: 语义重排 top_n=5（可选）
- 摘要注入策略：
  - 强制输出“来源依据”章节
  - 引用内容必须带 `document_id/chunk_id/page_no`

## 9. 迁移方案（P1）

### 9.1 迁移步骤
1. 新建表与索引。
2. 引入 Repository 层，先写入数据库并保留内存写（可选双写）。
3. 逐路由切换读取源至数据库。
4. 对账验证通过后移除内存态路径。

### 9.2 回滚策略
- 保留 feature flag：`USE_DB_PERSISTENCE`。
- 若异常，可一键切回内存读写（仅用于短时止损）。
