# 销售 AI 闭环系统 v0.0.2 —— 概要技术方案

## 1. 文档目的与范围
- 目标：承接 v0.0.2 需求文档，给出可落地的技术总体方案，指导后续详细设计与开发排期。
- 范围：聚焦 P0 RAG、P1 真实落库、P2 体验增强，不覆盖多租户与生产级高可用。
- 原则：优先保证链路可用与可追溯，其次保证兼容性，再逐步优化性能与体验。

## 2. 版本目标映射
- P0（FR-201~FR-203）：完成知识库上传、切片向量化、检索增强总结闭环。
- P1（FR-204~FR-205）：核心实体持久化，接口协议不变，重启后数据可恢复。
- P2（FR-206~FR-207）：分享页直开与合同结构化报价渲染。

## 3. 总体架构

### 3.1 逻辑分层
- 接入层：FastAPI 路由、请求校验、统一响应与错误映射。
- 领域服务层：RAG 服务、会话服务、PRD 服务、POC 服务、合同服务。
- 存储层：
  - 关系库：核心业务元数据。
  - 向量库：文档切片向量与检索。
  - 对象存储：PDF/音频文件原文。
- 前端层：上传管理、总结编辑、分享展示、合同下载。

### 3.2 关键技术选型
- 后端框架：FastAPI（延续现有实现，减少改造成本）。
- RAG 编排：LangChain。
- 向量数据库：Supabase PostgreSQL + pgvector。
- 持久化访问：Repository/DAO 模式，隔离路由与存储实现。
- 合同渲染：沿用 ReportLab，扩展模板字段映射能力。
- 前端：Next.js，新增分享页路由与知识库管理页模块。

## 4. 模块划分与职责

### 4.1 Knowledge 模块（P0）
- 负责 PDF 上传、文档解析、切片、embedding、索引写入、状态跟踪。
- 对外提供文档查询与检索接口。

### 4.2 Summary 模块（P0）
- 负责 transcript 聚合与 RAG 检索调用。
- 负责生成“带来源依据”的 PRD Markdown。

### 4.3 Persistence 模块（P1）
- 承接 users/sessions/utterances/prds/pocs/contracts/voice_profiles 的持久化。
- 提供 share_uuid、contract_id 等关键索引查询。

### 4.4 Share 模块（P2）
- 提供独立分享页的数据读取协议。
- 支持 URL 直入 UUID 自动渲染 Sandpack。

### 4.5 Contract 模块（P2）
- 将结构化报价字段注入合同模板。
- 保证报价字段来源可追溯到 PRD 或知识检索结果。

## 5. 核心业务流程

### 5.1 RAG 文档处理流程
- 上传 PDF -> 创建文档记录（pending） -> 异步切片与向量化（processing） -> 完成（ready）或失败（failed）。

### 5.2 总结增强流程
- 拉取会话 transcript -> 向量检索 TopK -> 构建增强 Prompt -> 生成 PRD -> 保存 PRD + 引用来源。

### 5.3 持久化迁移流程
- 先引入 Repository 层并保持接口协议不变 -> 双写验证（可选） -> 切主库读取 -> 下线内存存储。

### 5.4 分享直开流程
- 客户访问 `/share/{uuid}` -> 前端调用 `/poc/{uuid}` -> Sandpack 自动渲染 -> 异常时展示回退提示。

### 5.5 合同结构化渲染流程
- 读取 PRD 与报价字段 -> 注入合同模板 -> 输出 PDF -> 记录下载映射。

## 6. 数据架构概要
- 关系库核心表：users、sessions、utterances、prds、pocs、contracts、voice_profiles、knowledge_documents、knowledge_chunks。
- 向量索引：knowledge_chunks.embedding（pgvector）。
- 对象存储：音频原件与 PDF 原件仅保存 URL 与元数据。
- 关键索引：session_id、user_id、share_uuid、contract_id、document_status、created_at。

## 7. 接口设计概要
- 新增（P0）：
  - `POST /knowledge/upload`
  - `GET /knowledge/documents`
  - `POST /knowledge/reindex/{document_id}`
  - `POST /knowledge/retrieve`
- 改造（P0）：
  - `POST /session/{session_id}/summary` 增加检索增强与引用来源返回。
- 保持兼容（P1）：
  - v0.0.1 全量接口路径与主响应字段保持不变。

## 8. 非功能设计
- 可追溯：PRD 生成结果保留引用片段 ID、文档 ID、页码。
- 可观测：文档处理状态机、错误码、耗时指标、重试次数可查询。
- 安全：LLM/数据库密钥走环境变量；下载接口做资源存在与归属校验。
- 性能：P0 先以正确率为优先，提供可配置 TopK、chunk size、批处理大小。

## 9. 风险与缓解
- 向量召回不稳定：通过 chunk 策略、重排序、TopK 参数调优缓解。
- 数据迁移风险：以 Repository 抽象 + 灰度读写切换降低风险。
- 成本风险：embedding 与检索批量化、缓存热门文档检索结果。

## 10. 分阶段交付建议
- 里程碑 M1（P0）：RAG 最小闭环可用，summary 可输出来源依据。
- 里程碑 M2（P1）：核心数据持久化完成，重启恢复通过验收。
- 里程碑 M3（P2）：分享直开与合同结构化报价上线。
